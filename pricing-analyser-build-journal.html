Stage 1 · Discovery

Problem definition

Why does this need to exist, and for whom?
Done
Every product starts with a problem. But the sharpest problems come from people who have lived inside them — not read about them. The discovery for this project didn't begin at a whiteboard. It began in a series of situations, across several years and organisations, where the same structural gap kept surfacing.
The lived reality — not a designed persona Strategy professionals — MBA-trained consultants, commercial analysts, pricing managers — routinely face commercial questions that demand analytical rigour: how much margin is leaking through unmanaged discounts? Which sales reps are destroying pocket price? How should a SaaS packaging redesign affect ARR? These are not exotic questions. They arise in every commercial review, every pricing engagement, every GTM strategy project. And they consistently go under-answered — not for lack of intelligence or frameworks, but because the tools built to answer them are inaccessible to the people who need to ask them.
The access gap — who confirmed it

Before formalising the problem statement, the question was field-tested — not through structured surveys, but through the natural process of working alongside and consulting for commercial stakeholders across sectors. The signals were consistent.

Commercial director · Auto parts retail
Client-side commercial leader
Owned pricing and margin performance for a large retail network. Had access to SQL Server data but required a specialist analyst team and external consultants to run any meaningful pricing diagnostic. Turnaround for an ad-hoc leakage analysis: 2–3 weeks. By which point the pricing committee meeting it was prepared for had already happened.
Signal: "We know the problem is there. We just can't move fast enough to do anything about it before the next quarter."
Pricing manager · B2B distribution
In-house pricing function owner
Responsible for discount governance across a national sales force. Had a Tableau licence — but getting a new dashboard built required an IT request, a BI queue, and a 3-week turnaround. Ran the monthly governance review from an Excel model rebuilt by hand each month by a junior analyst who had since left the team.
Signal: "The model works, but only I know how it works. When she left, I had to start over."
Strategy consultant · Tier 1 firm
Engagement manager — pricing workstream
Led the analytical workstream on a pricing transformation. Had strong frameworks and hypotheses. Spent the first three weeks of the engagement negotiating data access with the client IT team, standing up the SQL environment, and validating the data joins — before a single analytical output could be produced for the client.
Signal: "By the time we had the waterfall built, the client had already formed their own view. We were playing catch-up with our own analysis."
Founder · Series B SaaS
Commercially-minded product leader
Strong product and commercial instincts. Faced a major packaging redesign decision — three potential tier structures, different metric options, significant ARR implications. Couldn't model it properly without a data engineer or a pricing consultant. Made the decision based on competitive benchmarking and intuition.
Signal: "I knew what I wanted to test. I just couldn't build the model fast enough to test it before we had to ship."
Problem statement
Formal problem statement
Strategy professionals have the frameworks to answer high-stakes commercial questions — but not the tools. The tool landscape offers a binary choice: enterprise software that requires procurement, implementation, and an IT team; or Excel rebuilt from scratch on every engagement. The gap between these two options is where most commercial analysis actually happens — and where the most value gets left on the table.
Confirmed across: pricing transformation engagements · in-house pricing functions · SaaS commercial design · strategy consulting delivery
Existing alternatives — why each fails this user
Out of reach
×SQL / Python / R — require engineering background not in the MBA or consulting toolkit
×Tableau / Power BI — expensive licences, BI team dependency, dashboard refresh lag
×Alteryx / Knime — workflow builder overhead, workspace generation, enterprise procurement
×Enterprise pricing software — six-figure contracts, months-long implementation, IT sign-off
What practitioners actually use
→Excel — rebuilt from scratch every engagement; collapses at scale; not reproducible
→Google Sheets — same limitations with worse performance
→Delegated to analysts — strategic context lost in translation; queue dependency
→Manual frameworks — estimated leakage, hand-drawn waterfalls, experienced guesses

Stage 2 · User Research

Validation

Does the problem hold up beyond lived experience?
Done
The discovery in Stage 1 was grounded in first-hand experience. Stage 2 validated that the problem isn't idiosyncratic — it is structural, recurring, and felt across the commercial strategy ecosystem. Validation came from three sources: direct observation on engagements, conversations with commercial stakeholders, and a teardown of the competitive tool landscape.
Validation source 1 — engagement observation
At a large consumer analytics function (2M+ customer base) The analytical infrastructure was well-resourced — SQL Server, dedicated analyst pools, external consultants for complex modelling. The data volumes and depth were appropriate for that infrastructure. But a structural friction persisted: the person who understood the business question well enough to know what to ask was not the person who could go and get the answer. Every analytical iteration — a changed hypothesis, a new segment cut, a follow-up question from the CFO — cost a queue position and a turnaround cycle. The exploratory work that produces the most valuable insights was the hardest to do at pace. On data governance: large volumes of customer and transactional data running through extended analyst pools and external consultants is a real and growing consideration — one that a client-side tool with no upload architecture directly addresses.
On a pricing transformation engagement (major US retailer) The analytical work required properly structured queries against client transaction systems — SQL, not Excel. Getting the environment live, the data access negotiated, and the connections validated consumed a significant portion of the early weeks. Once running, the analysis still went through specialist analysts. A question that surfaced in a client conversation at 4pm was not answerable before the following morning. The analytical pace was consistently behind the strategic pace. The insight — and the product motivation — was not that SQL is the wrong tool for production analytics. It is that the strategy practitioner needs to be able to run exploratory and presentation-ready analysis independently, at the speed of the conversation, without standing up infrastructure or joining a queue.
Validation source 2 — industry and consulting specialist signals
Conversations across three communities consistently surfaced the same friction pattern — not as a technology problem, but as an organisational and workflow problem.
A
Pricing specialists in consulting practices
Consistently reported spending 30–40% of early engagement weeks on data infrastructure setup rather than analytical output. The waterfall model, the price band regression, the governance dashboard — all rebuilt from scratch on every client. No standard tooling. No portable methodology.
Validation: the access gap is not a skills problem. It is a tooling gap that experienced practitioners feel just as acutely as junior analysts.
B
Commercial directors and CFO-adjacent roles
Pricing decisions in mid-market and enterprise organisations are often made in weekly or monthly commercial review cadences. The analytical support for those decisions arrives after the cadence — not in time for it. The gap between the question being asked and the analysis arriving is where decisions get made on instinct.
Validation: the demand is real and time-sensitive. Speed to insight is not a nice-to-have — it is the difference between analytical decision-making and gut-feel decision-making.
C
SaaS founders and commercial operators
Pricing decisions at Series A–C stage are among the highest-leverage commercial decisions available — and are consistently made without analytical support. The tools to do it properly are either too technical (Python-based conjoint modelling) or too expensive (specialist pricing consultants). Decisions default to competitive benchmarking and intuition.
Validation: the SaaS commercial design module has a clear and underserved market. No tool currently operationalises packaging design, wallet structuring, and ARR impact modelling for a non-technical founder.
Validation source 3 — competitor teardown

✓ Real-time data refresh

Stage 2 gate — passed The problem definition is confirmed externally, not just by lived experience. At least three practitioner communities — pricing consultants, commercial directors, SaaS operators — independently surface the same access gap. The competitive landscape confirms there is no middle-ground tool between Excel and enterprise software. The gap is real and unoccupied.

Stage 3 · Strategy

Product vision & strategic bets

What are we building, for whom, and why this, not something else?
Done
Vision statement
A strategy professional or commercial analyst should be able to go from raw business data to boardroom-ready diagnostics — pricing, revenue, margin, commercial architecture — without writing a line of code, without a Tableau licence, without a data engineering team, and without losing the strategic judgement that makes the numbers mean something.
Positioning
One-sentence positioning The Pricing Analyser is the commercial analytics workbench for strategy professionals who have always had the frameworks but never had the tools — built by a consultant who spent 12 years doing this work through infrastructure queues and analyst dependencies, and decided the gap was too important to leave unfilled.
Four strategic bets
1
Business brain, not engineering brain
Every design decision is optimised for someone who thinks in frameworks and business logic, not code or query syntax. No SQL. No Python. No workflow builders. No licence procurement. If a task requires a step that an MBA-trained professional wouldn't instinctively complete, the tool is responsible for abstracting it away.
This is not a dumbed-down version of a technical tool. It is a different kind of tool entirely.
2
The full analytical workflow — not just the output layer
DataBridge handles data ingestion, joining, cleaning, and transformation. The Pricing Analyser handles domain-specific analytics and diagnostics. Together they replace the full workflow that currently requires SQL (data prep), Python (analysis), Tableau (visualisation), and Excel (ad hoc manipulation) — for a user who has access to none of them.
3
Browser-native, zero infrastructure, data never leaves the machine
Every analytical operation runs client-side. No server, no database, no authentication, no upload. Pricing and commercial data is among the most sensitive data a business holds. A consultant on a client engagement cannot upload transaction data to a third-party server. The zero-server architecture is not a technical constraint — it is what makes the tool usable on real work.
4
Open-source consulting IP — the methodology is the product
The frameworks the tool operationalises come from real engagements across retail, technology, automotive, and media. Making this public is deliberate. The build-in-public format means every analytical decision is visible — the strategic reasoning behind it, not just the code. This is the credibility that a LinkedIn post about pricing strategy cannot manufacture.
What we are not building (non-goals)
Out of scope — v1.0
✕ Real-time data or live system connections
✕ User accounts or cloud data storage
✕ Multi-user collaboration
✕ Prescriptive price recommendations ("set $X")
✕ ERP / CRM / BI platform integrations
✕ Monetisation
Post-v1.0 backlog
→ CSV export templates for ERP handoff (v1.1)
→ Natural language finding summaries (v1.1)
→ Multi-currency normalisation (v1.1)
→ Saved session state via localStorage (v2.0)
→ Mobile-optimised layout (v2.0)

Stage 5 · Design & UX

Interface system

What does it look like and how does it feel to a non-technical practitioner?
Done
The most consequential stage before touching code. Every design decision filters through one test: would a strategy professional who has never written a line of code feel competent using this — or confused?
Six design principles
1
The framework is always visible
Every module names the analytical framework it implements — price waterfall, price band analysis — so the user understands what the output means, not just what the number is.
2
Configuration before computation
The user defines the analytical setup before the tool runs anything. Every result is traceable to a configuration the user made — they own the output because they built the inputs.
3
The output is the deliverable
Every module produces something a practitioner can put directly into a client presentation or CFO briefing — without reformatting. The tool ends where PowerPoint begins.
4
Data always stays in context
The user always knows what dataset is active, how many rows it contains, and what schema is applied. This is the trust signal — the equivalent of the filename in a spreadsheet.
5
Progressive disclosure
Default view shows the most important output. Advanced configuration is available but not surfaced until needed. The tool is not simple — but it starts simply.
6
One consistent layout across all 8 modules
Every module: configuration panel → output zone → export bar. Users learn the layout once. It never changes across modules.

Stage 6 · Technical Architecture

Build-ready technical specification

How is it built — every decision that constrains the code.
Done
Six locked architectural decisions
1
Single HTML file per tool — no build pipeline
Each tool is a self-contained HTML file. CSS inline, JS inline, libraries from CDN. No Webpack, no npm, no compilation. Opens and edits directly.
Matches DataBridge's architecture. Zero onboarding. Works on GitHub Pages. Deployable from any text editor.
2
Module system via JS objects — no framework
Each module implements a 4-method interface: init(), run(dataset, config), render(results), export(format). Registered in a central registry. Vanilla JS.
3
Session state in memory — single AppState object
All state lives in one in-memory AppState object. On page close, state is gone. No data persistence risk. The privacy guarantee is architectural, not a policy.
4
Web Workers for heavy computation
Any operation on more than 10K rows runs in a Web Worker. Main thread stays responsive. Waterfall aggregation, OLS regression, pivot tables — all off-thread.
NAPA-scale datasets: 100K+ transaction rows. Synchronous computation freezes the browser — unacceptable for a tool sold on speed.
5
CDN-only dependencies — version-pinned
PapaParse 5.4 · SheetJS 0.20 · Chart.js 4.4 · D3 7.9 · math.js 13 · jsPDF 2.5 · html2canvas 1.4. All loaded from cdnjs. No npm install required.
6
DataBridge handoff via sessionStorage
Schema-mapped dataset serialised to sessionStorage under key pa_incoming_dataset. Pricing Analyser reads on load, hydrates AppState, clears key. Tab-scoped. Never persists.
